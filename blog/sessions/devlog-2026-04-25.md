## Décisions

- **PostgreSQL boolean via PDO retourne des strings, pas des bools PHP.** `"f"` est truthy en PHP (string non-vide). Solution : comparer avec `=== true || === 't'` au lieu d'un cast implicite. Sans ça, `isAccessTokenRevoked()` retournait `true` pour tous les tokens, bloquant 100% des endpoints protégés.

- **CASCADE DELETE sur la table users.** Plutôt que de gérer manuellement la suppression des tokens/refresh_tokens liés, on ajoute une contrainte FK `ON DELETE CASCADE` dans une migration dédiée. Plus safe : impossible d'oublier un enfant orphelin.

- **UpdateProfileService séparé de CreateUserService.** Deux responsabilités distinctes, deux services distincts. Karpathy : flat > nested, fonctions courtes.

- **AuthFilter injecte l'user dans `AuthContext` (static holder).** Controller accède à `AuthContext::getUser()` sans dépendance sur la requête HTTP. Découplage service/transport.

- **C13 (DELETE /users/{id}) avec vérification d'ownership.** Même pattern que PUT profile : ownership check avant action, 403 si tentative sur un autre user.

---

## Impasses

- **Refresh token grant → `Token has been revoked` error.** Après activation du `RefreshTokenGrant`, les refresh tokens persistent en DB (`revoked=false` confirmé) mais la validation lève quand même l'exception. Cause probable : identifiant du token dans le JWT ne correspond pas à l'identifiant stocké (décryptage / claim mismatch). Non résolu dans cette session — bloqué à `⚠️`.

- **`RefreshTokenTrait` en conflit avec l'entity.** Le trait déclare les mêmes propriétés que `RefreshTokenEntity` qui les redéclare explicitement. PHP fatal. Fix : supprimer l'import du trait, garder les implémentations directes dans l'entity.

- **Migration `CreateUsersTable` avec colonne `password` au lieu de `password_hash`.** Le schéma prod utilise `password_hash`, les tests créaient une table avec `password`. Tests passaient en isolation mais les requêtes réelles échouaient (colonne inconnue). Fix : aligner le nom de colonne dans la migration de test.

## À ne pas oublier

> **PostgreSQL + PDO + PHP booleans = piège classique.**
> PDO retourne toujours des strings. `"f"` est truthy. Toujours comparer avec `=== 't'` ou caster via `filter_var($val, FILTER_VALIDATE_BOOLEAN)`. Ne jamais faire `if ($row['revoked'])` sur un champ boolean PostgreSQL.

> **Refresh token grant pas encore stable.** `POST /auth/token` avec `grant_type=refresh_token` lève `Token has been revoked` malgré un token valide en DB. Prochaine session : debugger le flow de décryptage du refresh token (identifier JWT claim vs DB identifier).
